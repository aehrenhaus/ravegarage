03 Existing steps

Step definitions are .net methods with an attribute on it. The attribute gives the match information.
I the later text, 'step' can mean either the 'step in feature file' or the 'step definition method' depending on the context.

There are common steps a well as steps only for domain(Rave).
You can find them in

  • Medidata.RBT.Common.Steps
  • Medidata.RBT.Features.Rave

Common steps can be used for any sites

  • I click "Commit" button
  • I navigate to url "http://www.bing.com"
  • I switch to "Targeted SDV Study Plan" window

    Although the value inside quotes is a rave thing, the step it self is indeed a common step.

Steps for Rave is Rave specific:

  • Then I verify that Tiers in subject override table are not in the following order
        | Tier Name | Row |
        |  All Forms | 1 |
        | No Forms | 2 |
  • And I select Report "Targeted SDV Configuration"

3 ways to learn about existing steps:

  • Read existing feature files.
  • Read step definition methods.
  • Read the step definition report in Medidata.RBT.Documents
    Run this project to update report , then open the report in Templates folder.

Guidelines for choosing a step

  • Prefer using existing steps than writing a new step.

    For example if you want to verify the study dropdown exists, use the common step:
    I verify control "Study" exists
    Don't write your step:
    I verify the study dropdown exists.

    To programmers:
    Step defs can be very adaptive because in most cases they are calling virtual methods on page object or interface methods under the hood.
    In the case that a step does a very similar thing but not excatly the what you want on a page, you can provide a override of such method on page object rather than writting a new step.


    For example:
    There is a common step: I choose "value" from "dropdown identifier"
    This step can choose text from a standard dropdown, but on your page the dropdown is a composite control.
    In this case , instead of write a new step I choose "value" from composite dropdown "identifier", you provide an override of ChooseFromDropdown(identifier, text) method on your page object.
    If the composite control appears on many pages of your site, you can override this method in a base class.
  • Prefer writing a common step def than writing a domain specific step, if you have to write a new step
    Just keep in mind that the step is common, others can share you method in similar situations. So often you will call a virtual or interface method rather than non-virtual one.
  • Finally, if you have to write a new domain specific step, the literal should be very specifc to describe your domain-specific problem. Don't let it looks like a common method.
